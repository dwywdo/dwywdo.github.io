자바의 프리미티브(Primitive) 타입 종류와 값의 범위, 그리고 기본 값

  • 자바는 변수에 대한 8가지 기본 타입을 제공하며, 이를 Primitive Types 이라고 부른다.
  • Primitive type은 Boolean, Character, 4개의 Integer, 2개의 Floating-point 타입으로 구성되어 있다.
  • Integer 타입과 Floating-point 타입은 모두 수를 표현하지만, 구성하고 있는 비트의 수/나타내는 방법 등이 다르다.
Type Contains Default Size Range
boolean true or false false 1 bit NA
byte Signed integer 0 8 bits -128 to 127
short Signed integer 0 16 bits -32768 to 32767
int Signed integer 0 32 bits -2147483648 to 2147483647
long Signed integer 0 63 bits -9223372036854775808 to 9223372036854775807
float IEEE 754 floating point 0.0 32 bits 1.4E-45 to 3.4028235E+38
double IEEE 754 floating point 0.0 64 bits 4.9E-324 to 1.7976931348623157E+308

프리미티브 타입(Primitive type) vs 레퍼런스 타입(Reference type)

  • 자바에서 제공하는 Reference type은 5가지가 있으며, 그 종류는 아래와 같다
    • Classes
    • Arrays
    • Interfaces
    • Enums
    • Annotations
  • 각각의 레퍼런스 타입에서는 별도로 다루는 것으로 하고, Primitive type과 어떤 차이가 있는지 확인해보자 (아래부터 Reference type의 인스턴스는 Object로 기술한다.)
    • Primitve type은 자바 언어에 의해서 정의된 것으로, 프로그래머는 새로운 Primitive type을 정의할 수 없다. 반면 Reference type은 사용자에 의해서 정의될 수 있으며, 그 종류에는 제한이 없다. 예를 들어, Reference type 중 하나인 Class의 하나로 Point라는 클래스를 생성하고 그 인스턴스를 이용할 수 있다.
    • Primitive type은 단일 값을 나타낸다. 반면 Reference type은 복수 개의 Primitive type을 포함하는 Aggregated type이 될 수 있다. 예를 들어 위에서 정의한 Point Class의 경우 2차원 상에서 x, y 좌표를 표현하는 두 개의 double type을 가질 수 있다. 이런 측면에서 char[]나 Point[]와 같은 Array 또한 Reference type임을 알 수 있다.
    • Primitve type은 0~8바이트까지의 메모리를 사용한다. Primitve type의 변수가 변수에 저장되거나 메소드에 전달될 때, 컴퓨터는 같은 값을 갖는 바이트를 복사해서 사용한다. 반면 Object는 메모리가 복사되는 것이 아닌, 원본 메모리에 대한 참조 값을 복사하여 사용한다. 또한 Object는 필요한 메모리에 따라 동적으로 힙에 저장된 뒤, 그 Object가 필요 없게 되면 Garbage Collection에 의해 메모리에서 자동으로 해제된다.
    • C/C++에서와 달리, 자바에서의 Reference type은 integer로부터 변환될 수 없으며 값 자체가 증가/감소될 수 없다 (메모리 주소에 대한 직접적인 연산이 불가능). 같은 맥락으로 주소 참조에 사용되는 operator인 &, *, -> 등도 없다.

Object 조작과 Reference Copies

  • 아래의 코드는 a primitive int value의 값을 변경한다. 
    int x = 42;
int y = x;
    • 변수 y는 변수 x의 값과 동일한 값을(복사된 값)을 가지게 된다. VM 상에서 두 개의 독립적인 32-bit integer 42가 존재하게 된다.
  • 같은 로직을 Reference type Point를 이용해보자. 
    Point p = new Point(1.0, 2.0);
Point q = p;
    • 위의 코드가 실행되고 나면, 변수 q는 변수 p와 동일한 참조 값(복사된 값)을 가지게 된다. 실제로 Point Object는 VM 상에서 하나만 존재하지만, 해당 Object에 대한 2개의 참조 copies가 존재하게 되는 것이다(p & q).
    • p와 q모두 Point Object의 값을 변경할 수 있다.
    • Arrays의 경우도 동일하게 생각할 수 있다. 
      char[] greetr = {'h','e','l','l','o'};
char[] cuss = greet; // cuss holds the same reference
cuss[4] = '!'; // Use reference to change an element
System.out.println(greet); // Prints "Hell!"
  • 메소드에 매개변수로 전달될 때에도 유사한 차이가 존재한다. 
    void changePrimitive(int x) {
    while(x > 0) {
        System.out.println(x--);
    }
}
  • 위의 메소드가 invoke되면, 메소드의 입장에서는 매개변수 x를 이용해 전달되는 변수의 복사본을 사용하게 된다. 
    void changeReference(Point p) {
    while(p.x > 0) {
        System.out.println(q.x--);
    }
}
  • 위의 메소드가 invoke되면, 메소드의 입장에서는 매개변수 p를 이용해 참조 값의 복사본을 사용하게 된다.
    • 실제 Object를 가리키는 참조 값을 사용하기 때문에, 메소드 내부에서 외부의 Point Object의 값(q.x)를 변경할 수 있게 된다. 하지만 p의 참조 값 자체를 변경할 수는 없다.

Comparing Objects

  • Reference type은 값의 동일 여부를 비교하는 데에도 Primitive type과 다른 점이 있다.
  • Primitive type의 값을 비교할 때에는, equality operator(==)를 사용한다. (값을 구성하는 모든 비트가 동일한지 비교)
  • Reference type의 값을 비교할 때에 ==를 사용하게 되면 참조 값을 비교하게 된다. (같은 Object를 가리키는지 비교) 
    String letter = "o";
String s = "hello";
String t = "hell" + letter;
if (s == t) System.out.println("equal") // They have same content, but they're not equal.

byte[] a = {1, 2, 3};
byte[] b = (byte[]) a.clone();
if (a == b) System.out.println("equal") // They're not equal.
  • Reference 측면에서 다른 두 String의 내용을 비교하고자 한다면, 메소드 String.equals(String s)를 사용한다. 
    String letter = "o";
String s = "hello"; // These two string objects
String t = "hell" + letter; // contain exactly the same text
if (s.equals(t)) { // And the equals() method
    System.out.println("equal"); // tells us so
}
  • 모든 object는 equals() 메소드를 상속받는다. (가장 상위 클래스인 Object로부터)
  • 동일성을 판단하는 비교 기준을 별도로 구현하고 싶다면 equals() 메소드를 오버라이딩하여 구현하면 된다.
  • String class는 이미 그에 맞게 구현되어 있기 때문에 별도로 구현하지 않고도 사용이 가능한 것이다.
  • 만약 Array에 대해서 equals()를 호출하게 되면, ==을 사용하는 것과 동일하다.
    • 그럼 어떻게 배열의 내용을 비교하나? java.util.Arrays.equals()를 사용한다.

Boxing & Unboxing Conversions

  • Primitive type의 값들을 object처럼 다루고 싶을 때가 있을 수 있다. 자바에서는 이러한 목적을 위해 Wrapper classes를 제공하며, 각 Primitive type에 대응되는 클래스들이 있다.
    • Boolean
    • Byte
    • Short
    • Character
    • Integer
    • Long
    • Float
    • Double
  • 위의 클래스들은 Immutable, final 클래스이며 단일 primitive value를 가질 수 있다.
  • 주로 이런 Wrapper 클래스들은 java.util.List 등의 Collection에 primitive value를 담고자 할 때 사용한다. 
    List<Integer> numbers = new arrayList<>();
numbers.add(new Integer(-1));
int i = numbers.get(0).intValue();
  • 자바는 Boxing / Unboxing Conversion을 통해 위와 같은 변환을 가능하게 한다.
  • Boxing: Primitive value -> Wrapper Object
  • Unboxing: Wrapper Object -> Primitive value
  • 별도의 타입캐스팅이 필요 없이, Authboxing/unboxing이 이루어진다. 
    Integer i = 0; // int literal 0 boxed to an Integer object
Number n = 0.0f; // float literal boxed to Float and widened to Number
Integer i = 1; // this is a boxing conversion
int j = i; // i is unboxed here
i++; // i is unboxed, incremented, and then boxed up again
Integer k = i+2; // i is unboxed and the sum is boxed up again
i = null;
j = i; // unboxing here throws a NPE(NullPointerException)
  • Autoboxing은 Collection을 더 쉽게 다룰 수 있게 해준다. 
    List<Integer> numbers = new ArrayList<>();
numbers.add(-1);
int i = numbers.get(0);

리터럴 (Literal)

Literal are sequences of source characters that diretly represent constant value that appear as-is in Java source code. – Evans, B., & Flanagan, D. (2018). Java in a Nutshell. Van Duuren Media.

  • 특정 문자들로 구성된 문자열이며, 그 자체로 상수 값을 소스 코드 내에서 표현한다.
    • 정수 값
    • 소수 값
    • single quote로 감싸진 단일 문자
    • double quote로 감싸진 string
    • 예약어인 true / false / null

변수 선언 및 초기화하는 방법

  • Type 변수의 이름 = 초기값 
    // 선언
int counter;
String s;
// 선언 및 초기화
int i = 0;
String s = readLine();
int[] data = {x+1, x+2, x+3}; // 배열 초기화
  • 자바 10에서 도입된 var를 사용하는 경우, 우항의 값에 따라서 자동적으로 타입은 추론된다 
    var i = 0; // type of i inferred as int
var s = readLine(); // type of s inferred as String

변수의 스코프와 라이프타임

  • 다른 언어에도 있는 개념이지만, 자바에서 선언되는 모든 변수는 Scope를 가지게 된다.

  • Scope는 특정 변수가 유효하게 접근되고 사용될 수 있는 프로그램의 어느 부분이라고 할 수 있다.

  • 자바에서의 Scope의 종류에는 다음과 같은 것들이 있다.

    • Class Scope: 클래스의 {} 안, 클래스 내 메소드의 밖에서 private access modifier와 함께 선언된 변수들은 Class Scope를 가진다. 이 변수들은 클래스 내의 어디에서도 사용될 수 있지만, 클래스 밖에서는 사용할 수 없다. 
      public class ClassScopeExample {
  private Integer amount = 0;
  public void exampleMethod() {
    amount++;
  }
  public void anotherExampleMethod() {
    Integer anotherAmount = amount + 4;
  }
}
      • private access modifier를 주지 않으면, 모든 패키지에서 해당 변수에 접근이 가능해진다. (Reference for Access Modifier in Java)
    • Method Scope: 메소드 내부에서 선언된 변수들은 Method Scope를 가지며, 동일한 메소드 내부에서만 접근이 가능하다. 
      public class MethodScopeExample {
  public void methodA() {
    Integer area = 2;
  }
  public void methodB() {
    area = area + 2;
    // compile error (cannot be resolved to a variable)
  }
}
    • Loop Scope: Loop (반복문) 내에서 선언된 변수들은 Loop Scope를 가지며, 해당 Loop 내에서만 접근이 가능하다. 
      public class LoopScopeExample {
List<String> listOfNames = Arrays.asList("Joe", "Susan", "Pattrick");
  public void iterationOfNames() {
      String allNames = "";
      for (String name : listOfNames) {
          allNames = allNames + " " + name;
      }
      String lastNameUsed = name;
      // compile error (cannot be resolved to a variable)
  }
}
    • Bracket Scope: {}를 별도로 선언하여 추가적인 Scope를 정의할 수 있다 
      public class BracketScopeExample {    
  public void mathOperationExample() {
      Integer sum = 0;
      {
          Integer number = 2;
          sum = sum + number;
      }
      // compiler error, number cannot be solved as a variable
      number++;
  }
}

  • 변수의 라이프타임은 기본적으로 Scope에 의존한다고 볼 수 있다.

    • 클래스로부터 생성된 인스턴스에 존재하는 변수(static access modifier 없이 선언된 변수)는 instance variable이라고 불리며, 인스턴스 생성으로부터 인스턴스가 할당된 메모리가 JVM에 의해 해제되는 순간까지가 Lifetime이다.
      • JVM에 의해 해제되는 순간은 해당 인스턴스 그 자체의 Scope에 또 의존한다.
    • 클래스에서 static과 함께 선언된 변수는 class variable이라고 불리며, 프로그램의 시작부터 종료까지가 lifetime이다.
    • 위의 두 가지 경우가 아니게 선언된 변수는 모두 local varaible이라고 볼 수 있으며, 프로그램의 제어가 해당 인스턴스가 선언된 블록({})을 벗어나게 되면 lifetime이 끝나게 된다.

  • 읽어보니 결국 주의할 점은 lifetime과 scope는 비슷하면서도 다른 개념이라는 것. lifetime의 경우는 말 그대로 해당 변수가 메모리를 할당받는 시점부터 끝날 때까지를 의미하는 것이고, scope의 경우는 그 변수의 lifetime 내에서 어디서 접근할 수 있는지를 의미한다고 정리해볼 수 있을 것 같다.

    • 예를 들어 Static 키워드를 사용해 선언된 Class variable의 경우에 lifetime은 프로그램 시작-종료까지이지만, private이 함께 지정되면 그 lifetime 내에서 해당 변수에 접근할 수 있는 범위는 클래스 내부로 제한되는 것. private이 없이 선언되면 어디서든 접근할 수 있게 된다.

스코프와 Variable Shadowing

  • 클래스 내에서 Class scope을 갖는 변수를 하나 선언하고, 그 클래스의 내부 메소드에서 동일한 이름으로 변수를 선언하면 어떻게 될까? 
    public class NestedScopesExample {
  String title = "Baeldung";
  public void printTitle() {
      System.out.println(title); // Print "Baeldung"
      String title = "John Doe";
      System.out.println(title); // Print "John Doe"
  }
}

타입 변환, 캐스팅 그리고 타입 프로모션

  • 자바의 특별한 Operator 중, Type Conversion or Casting에 쓰이는 것이 ()이다. 
    (byte) 28 // An integer literal cast to a byte type
(int) (x + 3.14f) // a floating-point sum value cast to an integer
(String)h.get(k) // A generic object to a String
  • 이 주제와 관련하여서는 Primtivie type과 Reference type을 분리하여서 보자.

Primitive types (Type Conversion)

  • 위에서 보았듯이, primitive type로 선언된 변수 각각은 구성하는 비트의 수. 즉 메모리의 크기가 다르다.
  • 보다 더 큰 범위를 갖는 타입 혹은 작은 범위를 갖는 타입으로 바꾸어 저장하는 것이 가능하고, 이를 Type Conversion이라 한다.
  • 보다 큰 범위로 Conversion할 때에는(Widening), 별도의 Notation이 코드 상에서 필요 없다. 남는 비트 공간의 경우 0으로 설정된다. 
    int myInt = 127;
long myLong = myInt; // int -> long conversion

float myFloat = myLong; // 정수 그룹 -> 소수 그룹
double myDouble = myLong; // 정수 그룹 -> 소수 그룹
  • 보다 작은 범위로 Conversion할 때에는(Narrowing), Conversion 대상의 값에 따라서 데이터가 손실될 수 있다. 이 경우에는 (), 즉 Casting을 명시적으로 해주어서 데이터 손실이 일어날 수 있는 상황임을 표현해야 한다 
    int myInt = (int) myDouble;
byte myByte = (byte) myInt;
  • 특수한 경우: Both of Widening and Narrowing
    • byte -> char로의 Conversion 케이스
    • byte -> int (Widening)
    • int -> char (Narrowing) 
      byte myLargeValueByte = (byte) 130; // 0b10000010 (-126)
char myLargeValueChar = (char) myLargeValueByte; // 0b11111111 10000010 (unsigned value)
int myLargeValueInt = myLargeValueChar; // 0b11111111 10000010 (65410)
byte myOtherByte = (byte) myLargeValueInt; // 0b10000010 (-126)

      char myLargeValueChar2 = 130; // This is an int not a byte! 0b 00000000 10000010 unsigned value
int myLargeValueInt2 = myLargeValueChar2; // 0b00000000 100000010 (130)
byte myOtherByte2 = (byte) myLargeValueInt2; // 0b10000010 (-126)
  • 위에서 소개한 Boxing / Unboxing의 경우도 Conversion의 한 종류에 해당한다.
  • String Conversions
    • 모든 Primitive types은 toString()을 이용해 String으로 변환할 수 있다. Overriding해서 커스텀하게 구현하는 것도 가능하다.
    • 이렇게 변환된 String을 다시 Primitive type으로 변경하고자 한다면, parse[Type] 메소드를 이용하면 된다. 
      byte  myNewByte   = Byte.parseByte(myString);
short myNewShort  = Short.parseShort(myString);
int   myNewInt    = Integer.parseInt(myString);
long  myNewLong   = Long.parseLong(myString);

float  myNewFloat  = Float.parseFloat(myString);
double myNewDouble = Double.parseDouble(myString);
  • Numeric Promotion
    • Binary operation을 수행할 때에, 각 피연산자는 데이터 크기 측면에서 아래의 규칙에 따라 Promotion 된다.
    • 두 피연산자 중 하나가 double이면, 다른 것은 double로 Promotion 된다.
    • 두 피연산자 중 하나가 float이면, 다른 것은 float로 Promotion 된다.
    • 두 피연산자 중 하나가 long이면, 다른 것은 long로 Promotion 된다.
    • 모든 경우가 아니면, 둘 다 int로 간주된다
    byte op1 = 4;
byte op2 = 5;
byte myResultingByte = (byte) (op1 + op2);
// op1, op2 모두 int로 Promotion된 후, 다시 byte로 캐스팅된다.

Reference types (Type Casting)

  • Reference type casting이 Primitive type conversion과 다른 점?
    • Primitive type conversion의 경우 데이터가 손실되는 경우 그 손실은 비가역적이다.
    • Reference type의 경우, 참조하고 있는 object 자체의 내용은 변경하지 않고, 단지 바라보는 방식만을 바꾸는 것이다.
      • Upcasting / Downcasting이라고 부른다
    • Upcasting: object에서 가용한 메소드와 속성을 축소한다
      • 상속과 밀접하게 연관이 되어 있으며, 주로 보다 구체적인 타입을 참조하기 위해 사용한다 
        public class Animal {
  public void eat() {
    //...
  }
}
public class Cat extends Animal {
  public void eat() {
    //...
  }
  public void meow() {
    //...
  }
}
Cat cat = new Cat();
Animal animal = cat; // Implicit upcasting
animal = (Animal) cat; // Explicit upcasting. 
// Explicit upcasting을 굳이 해줄 필요는 없다. 레퍼런스는 선언된 타입의 어떤 하위타입도 참조할 수 있기 때문에!
      • cat object는 그대로 존재함에도 불구하고, animal을 이용해 meow()를 호출하면 Compile error가 발생한다.
      • Upcasting의 장점? Polymorphism의 이점을 취할 수 있다. 다른 Dog class를 정의해보자. 
        public class Dog extends Animal {
  public void eat() {

  }
}
      • 모든 cat과 dog을 animal로 취급하는 feed() 메소드를 선언할 수 있다. 
        public class AnimalFedder {
  public void feed(List<Animal> animals) {
    animals.forEach(animal -> {
      animal.eat();
    })
  }
}
      • AnimalFeeder는 정확히 리스트 내의 동물이 어떤 동물인지 알 필요가 없다! Implicit upcasting은 List를 구성할 때 발생한다. 
        List<Animal> animals = new ArrayList<>();
animals.add(new Cat());
animals.add(new Dog());
new AnimalFeeder().feed(animals);
      • 각 Cat, Dog 클래스에서 eat()을 Overriding하면 먹는 방식을 다르게 구현하고, 일관된 방식으로 호출할 수 있다.
      • 참고로, 모든 자바의 object는 Object 클래스를 상속하기 때문에, 아래와 같은 Upcasting도 가능하다. 
        Object object = new Animal();
        • 모든 자바의 object가 toString()을 가지고 있는 이유이기도 하다.
      • Interface로 Upcasting하는 것도 흔한 케이스이다. 
        public interface Mew {
  public void meow();
}

public class Cat extends Animal implements Mew {  
    public void eat() {
        // ... 
    }
    public void meow() {
        // ... 
    }
}
        • 이제 어떤 Cat object는 Mew로 Upcasting할 수 있다. 
          Mew mew = new Cat();
        • Cat은 Mew이고, Implicit Upcasting이 일어난다.
        • 최종적으로 Cat object은 Animal, Object, Mew, Cat 타입의 Reference variable에 할당이 가능해진다.
    • Downcasting: object에서 가용한 메소드와 속성을 확장한다
      • 만약 Animal 타입을 Cat 클래스에만 존재하는 메소드(meow())를 호출하기 위해 사용하고 싶으면 어떻게 해야 할까?
      • Superclass -> Subclass로의 Casting
      Animal animal = new Cat();
((Cat) animal).meow();
      • 명시적으로(Explicitly) Casting을 해주어야 한다! (Upcasting과의 차이점)
      • 만약 animal에 할당된 대상이 Dog이거나, meow()를 가지고 있지 않은 클래스라면 없는 메소드에 접근하게 되므로 문제가 될 수 있다.
      • 이를 사전에 확인하기 위해, instanceOf operator를 사용해야 한다.
      if (animal instanceOf Cat) {
  ((Cat) animal).meow();
}
      • 만약 위의 instanceOf를 사용하지 않고 Downcasting을 하게 되면, 실제 Dog object를 캐스팅하려고 할 때 런타임 에러(ClassCastException)이 발생하게 된다.
      • 아예 관계없는 타입(상속 관계가 없는)으로 Downcasting하게 되면 컴파일 에러가 발생한다 
        Animal animal;
String s = (String) animal; // Cannot cast from Animal to String
    • cast() 메소드
      • 클래스를 캐스팅하기 위한 별도의 메소드가 있다. 
        public void whenDowncastToCatWithCastMethod_thenMeowIsCalled() {
  Animal animal = new Cat();
  if (Cat.class.isInstance(animal)) {
    Cat cat = Cat.class.cast(animal);
    cat.meow();
  }
}
        • 주로 Generic 타입과 함께 사용된다. Dog 혹은 Cat 중 한 가지 타입에 대해서만 AnimalFeeder를 정의해보자. 
          public class AnimalFeederGeneric<T> {
  private Class<T> type;

  public AnimalFeederGeneric(Class<T> type) {
      this.type = type;
  }

  public List<T> feed(List<Animal> animals) {
      List<T> list = new ArrayList<T>();
      animals.forEach(animal -> {
          if (type.isInstance(animal)) {
              T objAsType = type.cast(animal);
              list.add(objAsType);
          }
      });
      return list;
  }
} 

          @Test
public void whenParameterCat_thenOnlyCatsFed() {
    List<Animal> animals = new ArrayList<>();
    animals.add(new Cat());
    animals.add(new Dog());
    AnimalFeederGeneric<Cat> catFeeder
      = new AnimalFeederGeneric<Cat>(Cat.class);
    // 생성자를 통해서 어떤 동물에게만 먹이를 줄 것인지 결정하게 할 수 있다.
    List<Cat> fedAnimals = catFeeder.feed(animals);

    assertTrue(fedAnimals.size() == 1);
    assertTrue(fedAnimals.get(0) instanceof Cat);
}
